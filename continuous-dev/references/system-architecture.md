# system-architecture - 系统架构设计

**触发词**: `/system-architecture`, "架构设计", "系统架构"

**职责**: 系统级架构设计，服务划分，数据架构，部署方案

---

## When to Use

- domain-modeling 完成后，需要设计系统级架构
- 项目涉及多个服务/组件
- 需要明确技术选型和部署方案

**When NOT to Use:**
- 简单项目（单应用、单数据库）
- domain-modeling 还未完成

---

## 架构设计维度

### 1. 服务/组件划分

```
- 有哪些独立的服务/模块？
- 每个服务/模块的职责是什么？
- 服务之间的边界在哪里？

划分原则：
- 单一职责：一个服务做一件事
- 自治性：服务可以独立开发、部署
- 低耦合：服务间通过接口通信
```

### 2. 数据架构

```
数据存储选型：
- 关系型数据库（PostgreSQL/MySQL）
- 文档数据库（MongoDB）
- 键值存储（Redis）
- 消息队列（Kafka/RabbitMQ）

数据归属：
- 每个服务管理自己的数据
- 跨服务数据共享通过 API
```

### 3. 部署架构

```
部署方式：
- 单机部署
- 容器化（Docker/Kubernetes）
- Serverless
- 云原生

基础设施：
- 云服务商
- CI/CD 流程
- 监控告警
```

### 4. 接口设计

```
服务间通信：
- 同步（REST/gRPC）
- 异步（消息队列/事件）

API 设计原则：
- 宽进严出
- 版本管理
- 错误处理一致
```

### 5. 技术选型

```
语言/框架：每个服务用什么？为什么？
中间件：缓存策略、消息队列、搜索引擎
```

## 架构决策记录（ADR）

```markdown
# ADR-001: 选择消息队列

## 背景
服务间需要异步通信

## 选项
| 选项 | 优点 | 缺点 |
|------|------|------|
| Kafka | 高吞吐 | 运维复杂 |
| RabbitMQ | 轻量，易用 | 扩展性一般 |

## 决策
选择 RabbitMQ

## 理由
项目规模中小，RabbitMQ 足够，且运维简单
```
