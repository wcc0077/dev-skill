# checkpoint-verify - 检查点验收

**触发词**: `/checkpoint-verify`, "检查点", "验收"

---

## 核心原则

```
该用户确认的 → 交给用户确认
不该打扰的 → 不要打扰用户
```

**自动执行**：状态更新、下一步提示、文件写入
**等待确认**：human-verify、decision、human-action

---

## 3种检查点类型

| 类型 | 比例 | 执行方式 |
|------|------|----------|
| human-verify | 90% | 等待用户确认 |
| decision | 9% | 等待用户选择 |
| human-action | 1% | 等待用户操作 |

---

## 检查点格式（human-verify）

```
╔═══════════════════════════════════════════════════════╗
║  CHECKPOINT: Verification Required                    ║
╚═══════════════════════════════════════════════════════╝

Task: [任务名]

Built: [Claude 自动化的成果]

How to verify:
  1. [具体验证步骤]
  2. [预期结果]

────────────────────────────────────────────────────────
→ YOUR ACTION: Type "approved" or describe issues
────────────────────────────────────────────────────────
```

**用户确认后** → 自动进入下一阶段

---

## decision 格式

```
╔═══════════════════════════════════════════════════════╗
║  CHECKPOINT: Decision Required                       ║
╚═══════════════════════════════════════════════════════╝

Decision: [什么需要决定]

Options:
  1. [选项A] — 优点/缺点
  2. [选项B] — 优点/缺点

────────────────────────────────────────────────────────
→ YOUR ACTION: Select [option-id]
────────────────────────────────────────────────────────
```

**用户选择后** → 自动进入下一阶段

---

## 自动进入下一阶段

当检查点通过时：

```
1. 自动更新 PROJECT_STATUS/data.md
2. 自动显示下一阶段提示（不打断用户）
3. 用户可以继续，也可以去做其他事
```

**示例**（检查点通过后的自动提示）：
```
✅ 检查点通过

自动更新：PROJECT_STATUS/data.md 已更新
下一步：可以运行 /parallel-implement 继续

（如果用户没有其他问题，不会再打扰）
```
