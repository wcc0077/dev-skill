# idea-research - 想法调研

**触发词**: `/idea-research`, "调研一下", "帮我分析这个想法"

**职责**: 系统性调研想法，拆解为可执行步骤

---

## When to Use

- 用户有一个想法，需要评估可行性
- 需求模糊，需要拆解为具体步骤
- 技术选型不确定，需要调研方案

**When NOT to Use:**
- 需求已经非常清晰，只差实现
- 单个小任务（<1小时）

---

## 意图澄清（可选）

如果想法模糊，先澄清再调研。

**判断标准**：
```
□ 能用一句话说清楚要做什么吗？
□ 能说出"用户是谁"吗？
□ 能说出"期望结果"吗？

如果任何一个 ❌ → 先进入意图澄清
```

**澄清策略**：一次一问，多选项优先（借鉴 brainstorm）

```
❌ 开放式："你想要什么样的用户系统？"
✅ 多选项："用户系统需要支持哪些功能？
   A. 注册/登录/登出
   B. 第三方登录
   C. 角色权限管理
   D. 以上都要
   "
```

最多问 3-5 个问题。

## 调研方法论（时间盒：30分钟）

### 搜索优先级

```
1. GitHub 搜索     → 类似项目、解决方案
2. 库/框架 文档    → 官方能力边界
3. npm/pypi/crates → 现有工具
4. Exa/Web 搜索   → 论文、博客
```

**原则**：找到 80% 相似方案后停止搜索

---

## 声明假设

在开始调研前，先声明假设并让用户确认：

```
ASSUMPTIONS I'M MAKING:
1. 这是 Web 应用（不是移动端/桌面端）
2. 使用 [已有技术栈]
3. 项目目标是 [上线/内部使用/POC]
→ 如果这些假设不对，请纠正我
```

---

## 调研维度

**维度1：本质是什么**
- 要解决的根本问题是什么？（1句话）
- 不解决这个问题会怎样？（代价）
- 用户当前怎么解决？（现状）

**维度2：技术路径**
- 方案A：... → 适合...，不适合...
- 方案B：... → 适合...，不适合...
- 推荐：方案X，理由...

**维度3：实现代价**
- 最小可行方案：...（1-2天）
- 完整方案：...（2-4周）

---

## 不确定性量化

```
用概率表达不确定性，不要用"可能"/"也许"

❌ 可能是用Redis做缓存
✅ 80%用Redis，20%需要探索其他方案

量化标准：
- ≥90%：很确定，实践中按此执行
- 70-89%：基本确定
- 50-69%：不确定，需要原型验证
- <50%：高风险，优先做POC
```

```markdown
## 技术路径
- 推荐方案：...
- 备选方案：...

## 实现步骤
| 步骤 | 依赖 | 预计工期 | 确定性 |
|------|------|----------|--------|

| 2.   | 1    | 1天      | 70%    |

## 风险点
| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| X    | 20%  | 高   | ...      |

## 可复用资源
- GitHub: ...
- 库: ...
```

## 验收标准

```
□ 根本问题是否用1句话说清楚？
□ 技术路径是否有2个以上选项？
□ 步骤是否独立可执行？
□ 风险是否量化（概率+影响）？
□ 工期估算是否分解到天？
```

## Red Flags

| 信号 | 含义 |
|------|------|
| 想法超过 15 年经验无法拆解 | 可能需要先做 POC |
| 所有步骤的确定性都 <50% | 想法太模糊 |
| 工期估算超过 4 周 | 应该拆分 |
