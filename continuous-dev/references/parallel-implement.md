# parallel-implement - 并行实现

**触发词**: `/parallel-implement`, "开始实现", "实现契约"

**职责**: 基于契约的依赖分析，并行实现，TDD循环

---

## When to Use

- 契约文档已生成，需要开始实现
- 需要多人/多 agent 并行开发

**When NOT to Use:**
- 契约还未生成或未验证
- 单个简单任务，不需要并行

---

## Task Sizing

| 级别 | 范围 | 示例 | 工期 |
|------|------|------|------|
| **XS** | 1 个函数 | 验证器、正则 | <2h |
| **S** | 1-2 个文件 | 单个 API | 0.5-1天 |
| **M** | 3-5 个文件 | 单个功能 | 1-3天 |
| **L** | 5-8 个文件 | 跨多组件 | 需并行 |
| **XL** | 8+ | **太大，需拆分** | - |

> Agent 在 S 和 M 任务表现最好

---

## 依赖分析（必须先做）

```
□ 列出所有 Operation
□ 识别每个 Operation 的依赖项
□ 确定哪些可以并行，哪些必须串行
□ 识别共享状态和竞争条件
```

### 并行可行性判断

```
可以并行：
✅ 读不同的实体
✅ 写不同的实体
✅ 异步副作用（消息队列）

不能并行：
❌ 写同一个实体
❌ 读-写同一个实体
❌ 竞争共享资源
```

---

## TDD 循环

```
RED    → 写一个失败的测试
GREEN  → 写最小代码让测试通过
REFACTOR → 改善代码，保持测试绿
```

---

## 实现原则

1. **实现跟随契约** — 不猜测，不添加契约没有的内容
2. **外部状态用 mock** — 数据库、API、消息队列都用mock
3. **幂等性** — 操作可重复执行，结果一致
4. **错误处理** — 对应错误契约的每个 Error

---

## 覆盖率要求

```
□ 单元测试覆盖率 ≥ 80%
□ 契约的每个 postcond 都有测试覆盖
□ 契约的每个 error 都有测试覆盖
□ 关键路径覆盖率 100%
```

## 下一步

实现完成后 → **自动触发** `/spec-compliance-review`

**自动化行为**：
- 自动更新 PROJECT_STATUS
- 自动运行测试套件
- 覆盖率≥80% 时自动进入审查阶段

**需要用户确认**：检查点 CP4（覆盖率 ≥ 80%）
- 回复"继续"或空 → 进入契约审查
- 回复"补充测试" → 留在实现阶段

## Red Flags

| 信号 | 含义 |
|------|------|
| 依赖分析发现竞争条件 | 依赖分析有遗漏 |
| 集成测试失败 | 契约定义有歧义 |
| 覆盖率达不到 80% | 契约设计可能有问题 |
